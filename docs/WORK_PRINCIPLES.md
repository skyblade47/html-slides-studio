## 原则零：功能对齐检查 — 所有改动必须符合产品定位

### 0.1 核心理念

> **任何代码修改、功能设计、架构调整，必须首先通过产品定位一致性检查。发现偏离时必须取得用户确认才能继续推进。**

### 0.2 产品核心定位（不可偏离）

| 维度 | 定义 | 来源 |
|------|------|------|
| **产品名称** | HTML 演示文稿工作室 | `README.md` |
| **产品定位** | AI 驱动的 HTML 演示文稿生成工具 | `README.md` |
| **核心哲学** | 单 HTML 文件输出，无需依赖，方便分享 | `README.md` |
| **技术栈** | Tauri + React + TypeScript + Vite | 项目配置 |
| **输出格式** | 自包含的 HTML 文件，支持所有浏览器 | 核心特性 |

### 0.3 明确禁止的方向（除非用户明确批准）

以下方向与产品核心定位冲突，**默认禁止**，必须取得用户明确确认后方可推进：

| 禁止方向 | 冲突原则 | 如需推进 |
|---------|---------|---------|
| 添加后端服务器依赖 | 违背"单文件输出" | 用户书面确认 |
| 打包为可执行文件 | 违背"HTML传输"理念 | 用户书面确认 |
| 使用商业组件库（需授权） | 违背开源免费原则 | 用户书面确认 |
| 引入大型第三方库 | 违背"轻量级"定位 | 用户书面确认 |
| 移除 TypeScript 类型系统 | 违背代码质量标准 | 用户书面确认 |

### 0.4 功能对齐检查清单

每次开始任何开发任务前，**必须先完成此检查清单**：

```
□ 这项改动是否符合"HTML演示文稿生成"的核心定位？
  → 是否保持了单文件输出能力？
  → 是否便于在浏览器中打开？

□ 这项改动是否符合"轻量级"的设计原则？
  → 新增依赖是否必要？
  → 打包体积是否可控？

□ 这项改动是否在"不应引入的功能"清单中？
  → 参考 FINAL_WORK_CHECKLIST.md "不应引入的功能"表

□ 这项改动是否仅针对演示文稿生成场景？
  → 产品定位是"HTML演示文稿生成工具"

□ 如以上任何一项回答为"否"或"不确定"：
  → ⚠️ 立即暂停，向用户说明偏离点
  → 等待用户明确确认后方可继续
```

### 0.5 偏离确认流程

```
检测到可能偏离 → 整理偏离分析 → 提交用户确认 → 根据用户决策执行
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
       批准推进    修改方案     否决放弃
          │           │           │
     记录决策    重新对齐    记录决策
     继续工作    再次检查    关闭任务
```

**偏离分析报告格式**：

```markdown
## 功能对齐偏离提醒

**任务**: [任务名称]
**偏离点**: 具体哪些设计不符合产品定位
**风险等级**: 高/中/低
**影响范围**: 可能影响哪些现有功能或原则
**替代方案**: 如果有更符合定位的实现方式
**建议**: 推进 / 修改方案 / 放弃

请用户确认决策。
```

## 原则一：设计方案 + 精简代码

### 1.1 核心理念

> **设计先行，代码后行。用最精简的方式实现功能——最好的代码是你没写的代码。**

### 1.2 精简检查清单（设计评审阶段使用）

```
□ 这个功能能否用更少的代码实现？
□ 是否引入了不必要的抽象层？
□ 能否复用现有代码而非新建？
□ 新增的依赖是否真的必要？
□ 是否过度设计了（为未来写代码）？
□ 能否用一个函数替代一个类？
□ 能否用一行配置替代一个模块？
```

### 1.3 五阶段详解

| 阶段 | 产出物 | 检查点 |
|------|--------|--------|
| **1. 需求分析** | 功能需求文档 | 目标清晰、范围明确 |
| **2. 设计方案** | 技术方案 + 架构图 | 接口定义、数据流、组件树 |
| **3. 精简检查** | 精简后的方案 | **最少代码、最少文件、最少依赖** |
| **4. 设计评审** | 评审记录 + 修改意见 | 可行性、一致性、影响范围 |
| **5. 编码实现** | 代码 + 测试 | 遵循精简方案 |
| **6. 验证收尾** | 检查报告 | 功能/类型/构建全通过 |

### 1.4 设计方案必须包含

```
设计方案模板：
□ 功能概述（一句话说清楚做什么）
□ 涉及文件清单（新增/修改哪些文件）
□ 接口/类型定义（Props、Store interface）
□ 数据流图（数据从哪来、到哪去、怎么变）
□ 组件树（UI 层级结构）
□ 边界情况（空状态、错误状态、加载状态）
□ 影响范围（是否影响现有功能）
```

### 1.5 设计阶段禁止事项

- ❌ 跳过设计方案直接写代码
- ❌ 设计方案中没有组件树就开始编码
- ❌ 不评审就进入编码
- ❌ 编码时随意偏离设计方案（如需偏离，先更新方案）

## 原则二：编写 + 检查（写→审→改→查闭环）

### 2.1 核心理念

> **同一个 AI 不能既当运动员又当裁判。编写和检查必须由不同的 subagent 完成。审查者只给意见不直接修改，形成写→审→改→查的完整闭环。**

### 2.2 四轮闭环审查流程

```
                        ┌──────────────────────────────────────┐
                        │           第一轮：编写 (Write)         │
                        │                                      │
                        │  ┌─────────────────────────────────┐ │
                        │  │  Coder (编写者)                 │ │
                        │  │  - 实现功能                     │ │
                        │  │  - 遵循设计方案                 │ │
                        │  │  - 添加注释                     │ │
                        │  └──────────────┬──────────────────┘ │
                        │                 │ 产出：代码         │
                        └─────────────────┼────────────────────┘
                                          ▼
                        ┌──────────────────────────────────────┐
                        │          第二轮：审查 (Review)         │
                        │                                      │
                        │  ┌─────────────────────────────────┐ │
                        │  │  Reviewer (审查者)               │ │
                        │  │  - 逐条检查代码                 │ │
                        │  │  - ⚠️ 只给修改意见，不动代码    │ │
                        │  │  - 输出审查报告                 │ │
                        │  └──────────────┬──────────────────┘ │
                        │                 │ 产出：审查意见     │
                        └─────────────────┼────────────────────┘
                                          ▼
                              ┌───────────────────┐
                              │  审查是否通过？     │
                              └───────┬───────┬───┘
                                 通过 │       │ 不通过
                                      │       ▼
                                      │ ┌──────────────────────────────────────┐
                                      │ │        第三轮：修复 (Fix)              │
                                      │ │                                      │
                                      │ │  ┌─────────────────────────────────┐ │
                                      │ │  │  Coder (编写者)                  │ │
                                      │ │  │  - 根据审查意见逐条修复         │ │
                                      │ │  │  - 标注修复内容                 │ │
                                      │ │  └──────────────┬──────────────────┘ │
                                      │ │                 │ 产出：修复代码     │
                                      │ └─────────────────┼────────────────────┘
                                      │                   │
                                      │                   ▼
                                      │ ┌──────────────────────────────────────┐
                                      │ │        第四轮：复查 (Verify)           │
                                      │ │                                      │
                                      │ │  ┌─────────────────────────────────┐ │
                                      │ │  │  Reviewer (审查者)               │ │
                                      │ │  │  - 逐条验证修复结果             │ │
                                      │ │  │  - ⚠️ 仍只给意见，不动代码      │ │
                                      │ │  │  - 输出复查报告                 │ │
                                      │ │  └──────────────┬──────────────────┘ │
                                      │ │                 │                    │
                                      │ └─────────────────┼────────────────────┘
                                      │                   │
                                      │      ┌────────────┴────────────┐
                                      │      ▼                         ▼
                                      │   修复通过                  修复不通过
                                      │      │                         │
                                      └──────┘              (返回第三轮，最多3次)
                                                              │
                                                              ▼
                                                         超过3次上报
                                                         Main Agent 介入
                                      │
                                      ▼
                        ┌──────────────────────────────────────┐
                        │         Main Agent 最终确认           │
                        │  - 合并结果                          │
                        │  - 记录经验                          │
                        │  - 更新错误库                        │
                        └──────────────────────────────────────┘
```

### 2.3 核心铁律：审查者不直接修改代码

> ⚠️ **审查者永远只输出审查意见，绝不直接修改代码。**

| 审查者可以 | 审查者不可以 |
|-----------|-------------|
| ✅ 指出具体问题位置（文件+行号） | ❌ 直接修改任何代码文件 |
| ✅ 说明问题原因和影响 | ❌ 运行 `replace_in_file` / `write_to_file` |
| ✅ 给出修复建议（可附示例代码片段） | ❌ 以"帮你改好了"的方式操作 |
| ✅ 标注严重程度（严重/中等/轻微） | ❌ 在意见中使用模糊表述（"可能有问题"） |
| ✅ 在复查时逐条对照确认修复 | ❌ 复查时发现新问题直接修改 |

### 2.4 审查检查清单

```
审查检查清单：
□ 类型定义是否完整（无隐式 any）
□ Props 接口是否完整
□ useEffect cleanup 是否正确
□ 定时器/监听器是否正确销毁
□ 错误处理是否覆盖所有分支
□ 是否引入不必要的依赖
□ 是否影响现有功能
□ 代码风格是否符合项目规范
□ React 组件是否遵循 hooks 规则
□ CSS 类名是否遵循规范

审查报告格式：
```markdown
# 审查报告 - [任务名称]

## 总体评价
[通过 / 需要修复]

## 问题列表

### [严重] 问题1：xxx
- **位置**: `path/file.tsx:行号`
- **问题**: 具体描述
- **影响**: 可能导致的后果
- **建议**: 修复方案（可附示例代码片段）

### [中等] 问题2：xxx
- ...

### [轻微] 问题3：xxx
- ...
```
```

### 2.5 审查不通过的处理

```
第一次审查不通过
    ↓
编写者根据审查报告修复（标注每处修复对应的问题编号）
    ↓
审查者复查（逐条验证，仍只给意见）
    ↓
  ┌─ 通过 → 提交 Main Agent
  └─ 不通过 → 编写者再次修复 → 审查者再次复查
                  ↓ (第3次仍不通过)
              Main Agent 介入，人工决策
```

**循环上限**：最多 3 轮修复-复查。超过则上报 Main Agent，防止无限循环。

## 原则三：错误库建立

### 3.1 核心理念

> **同样的错误不犯第二次。每个错误都是团队的资产。**

### 3.2 错误库结构

错误库文件：`docs/ERROR_LIBRARY.md`

```markdown
# 📋 错误库

## 错误分类

### 类型错误
### 运行时错误
### 逻辑错误
### 规范违反
### 架构问题
```

### 3.3 错误记录模板

每条错误记录必须包含：

```markdown
### [E-XXX] 错误标题

| 字段 | 内容 |
|------|------|
| **错误ID** | E-001 |
| **发现日期** | 2026-06-05 |
| **严重程度** | 严重 / 中等 / 轻微 |
| **涉及文件** | `path/to/file.tsx` |
| **错误类型** | 类型错误 / 运行时错误 / 逻辑错误 / 规范违反 |
| **错误描述** | 一句话描述错误现象 |
| **错误原因** | 根因分析 |
| **修复方案** | 具体修复步骤 |
| **预防措施** | 如何避免再次发生 |
| **相关模式** | 关联的代码模式 ID |
```

### 3.4 常见错误分类（HTML 演示项目）

| 分类 | 典型错误 | 预防措施 |
|------|---------|---------|
| **React 错误** | useEffect 依赖缺失、状态更新滞后 | 代码模式参考 |
| **TypeScript 错误** | 隐式 any、Props 不完整 | `npm run typecheck` 阻断 |
| **构建错误** | 依赖未安装、路径错误 | 检查 package.json |
| **HTML 输出错误** | 动画类名冲突、CSS 变量未定义 | 输出前预览检查 |
| **状态管理错误** | Zustand store 初始化问题 | 遵循 store 规范 |
| **组件错误** | 组件未正确导出、Props 类型错误 | 代码审查 |

## 原则四：可复用代码库 + 代码参考库

### 4.1 核心理念

> **不要重复造轮子。写好一次，到处使用。参考优质代码，站在巨人肩膀上。**

### 4.2 代码库目录结构

```
src/
├── components/          # React 组件
│   ├── Editor/         # 编辑器相关组件
│   ├── Preview/         # 预览相关组件
│   └── Layout/          # 布局组件
│
├── pages/              # 页面组件
│
├── services/           # 服务层
│   ├── templates.ts    # 模板生成
│   ├── ai.ts           # AI 服务
│   └── animationEngine.ts  # 动画引擎
│
├── stores/             # 状态管理
│   ├── editorStore.ts  # 编辑器状态
│   └── projectStore.ts # 项目状态
│
├── types/              # TypeScript 类型定义
│   └── index.ts        # 类型导出
│
└── styles/             # 样式文件
    └── index.css       # 全局样式
```

### 4.3 可复用代码质量标准

**可复用代码必须满足**：
- [ ] 完整的 TypeScript 类型定义
- [ ] 中文 JSDoc 注释
- [ ] 无外部副作用
- [ ] 单一职责
- [ ] 边界情况处理

## 快速检查清单

### 每次开发任务开始前，确认：

```
□ 是否通过了功能对齐检查？（原则零）
□ 若有偏离产品定位，是否已取得用户确认？（原则零）
□ 是否有设计方案？（原则一）
□ 设计方案是否经过评审？（原则一）
□ 是否计划使用 Coder + Reviewer 双 subagent？（原则二）
□ 是否查阅了错误库中的相关记录？（原则三）
□ 是否有可复用的现有代码？（原则四）
```

### 每次开发任务完成后，确认：

```
□ 是否更新了错误库（如有新错误）？（原则三）
□ 是否有新的可复用代码需要沉淀？（原则四）
□ Reviewer 的检查清单是否需要更新？（原则二+三）
□ TypeScript 检查是否通过？（npm run typecheck）
□ 项目构建是否成功？（npm run build）
```
